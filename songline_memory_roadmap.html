SonglineMemory development roadmap

Phase 1 — Core ✓ Complete
Two-pass context compressor
Pivot-based segmentation + TF-IDF sentence scoring. Strips dictation noise, extracts core facts, outputs (concept_label, core_data) tuples ready for add_landmark(). Pivot leak fix included.
Core memory schema — two-database architecture
landscape.db (active graph) and lore.db (long-term wiki) — cleanly separated. Full SongKeeper hooks baked in: memory_tier, relevancy_score, recall_count, unique_query_count, walk_count, strength_score.
Auto-weave Songline edges on write
Every new Landmark scores against all active Landmarks via TF-IDF cosine similarity. Top neighbors get template-based narrative edges woven on arrival. The graph builds itself.
TF-IDF retrieval with relevancy boost
Query walks the active landscape via cosine similarity. Proven memories get a mild relevancy multiplier. Returns ranked Landmarks with full Songline narrative paths attached.
Duplicate landmark detection
Jaccard similarity check at 0.80 threshold before every placement. Catches duplicates against the database AND within the same batch import. Keeps the landscape clean.
Write pipeline with source tagging
Single entry point for all memory writes — live session, batch import, or agent-generated. Source tag (e.g. "gemini_wiki") prefixes concept labels for Lore queryability by origin.
Phase 2 — SongKeeper ✓ Complete
Songline strength scoring and decay
Every traversal strengthens an edge (+0.05, capped at 1.0). Unwalked edges decay each SongKeeper cycle (-0.05). Edges at or below 0.10 are pruned. The graph reflects actual use.
Relevancy scoring engine
Three weighted signals: recall frequency (40%), unique query diversity (35%), recency (25%). Recalculated every SongKeeper cycle. Scores drive promotion gate decisions.
Lore promotion with triple-gate system
Landmarks must pass all three gates to reach Lore: relevancy ≥ 0.6, recall ≥ 3, unique queries ≥ 2. Not majority — all three. Only memories proven across multiple contexts survive.
DREAMS.md — Dream Diary
Every SongKeeper cycle appends a human-readable entry: what decayed, what was pruned, what relevancy scores shifted, what was promoted to Lore. Trust through transparency.
Cron scheduling on 851Office-1
SongKeeper runs nightly at 2am via cron. No manual intervention. Logs to songkeeper.log. Runs whether you're at the machine or in Florida.
