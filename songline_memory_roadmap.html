SonglineMemory development roadmap

Phase 1 — Core (start here)
Build the retrieval function
Walk the Songlines from a query — find relevant Landmarks by concept label, return the connecting narrative path. This is the open loop Gemini left.
Auto-generate Songline edges on write
When a new Landmark is placed, auto-generate a narrative connection to recent/related Landmarks. The "weave" step — core to the whole concept.
Replace hash compressor with semantic similarity
Swap out the hashing algo for something that preserves meaning, not just identity. Pure Python, zero external deps — TF-IDF or cosine similarity via math module.
Phase 2 — Dream cycle
Songline strength scoring
Track how often each edge is walked. High-traffic connections strengthen. Low-traffic ones decay. The differentiator — nobody else does this on a graph.
Dream consolidation pass
Scheduled pass that prunes orphaned Landmarks, merges near-duplicates, and rewrites weakened Songlines. Inspired by OpenClaw's 3-phase + scoring gates.
Dream diary (DREAMS.md equivalent)
Human-readable log of what the consolidation pass did. Trust through transparency — OpenClaw's best UI idea, adapted for our system.
Historical backfill
Feed old notes/context retroactively through the landscape. Lets new models onboard to existing memory without manual re-entry.
Phase 3 — Multi-model & publish
Multi-model API layer
Expose the landscape via a local API endpoint. Claude, Gemini, and Gemma all read/write to the same SQLite. Shared operational memory across models.
Clean up README and repo for public attention
Rewrite docs, add usage examples, publish the differentiated positioning. Target AI/open source community on GitHub and HN.
